# 一、DLL 相关内容

## 1.1 DLL 基础概念

### 什么是 DLL

**DLL（Dynamic Link Library，动态链接库）** 是一种包含代码和数据的可执行文件格式，允许多个程序同时使用同一块代码和数据。

### DLL 与静态库的区别

| 特性 | 静态库 (.lib/.a) | 动态库 (.dll/.so) |
|------|-----------------|-------------------|
| 链接时机 | 编译时链接 | 运行时链接 |
| 代码复制 | 每个程序一份 | 多个程序共享一份 |
| 文件大小 | 可执行文件较大 | 可执行文件较小 |
| 更新方式 | 需要重新编译 | 只需替换 DLL |
| 内存占用 | 每个进程独立占用 | 多进程可共享内存 |
| **加载方式** | 隐式加载（编译时） | 显式/隐式加载 |

### 跨平台动态库命名

| 平台 | 动态库扩展名 | 静态库扩展名 |
|------|-------------|-------------|
| Windows | .dll | .lib |
| Linux | .so | .a |
| macOS | .dylib | .a |

### DLL 的优势

1. **代码复用**：多个程序可共享同一 DLL
2. **模块化**：便于分模块开发和维护
3. **节省内存**：多个进程共享同一份代码页
4. **易于更新**：只需替换 DLL 文件，无需重新编译主程序
5. **插件架构**：支持运行时扩展功能

## 1.2 DLL 的工作原理

### 导出表与导入表

```cpp
// 【面试重点】DLL 的核心机制

// 导出表（Export Table）：DLL 中可被外部使用的符号列表
// - 存储在 DLL 的 PE（Portable Executable）文件头中
// - 包含函数名、序号、地址等信息

// 导入表（Import Table）：EXE/DLL 需要从其他 DLL 导入的符号列表
// - 存储在加载该 DLL 的模块中
// - Windows 加载器根据导入表解析依赖关系
```

### 名称修饰（Name Mangling）

```cpp
// C++ 支持函数重载，编译器需要对函数名进行编码

// 源代码
void func(int) {}
void func(double) {}

// 编译后可能变成（VC++）：
// ?func@@YAXH@Z        // void __cdecl func(int)
// ?func@@YAXN@Z        // void __cdecl func(double)

// 【面试重点】解决方案：extern "C"
extern "C" {
    // 使用 C 链接约定，不进行名称修饰
    __declspec(dllexport) void plain_func() {}
    // 导出后函数名为：plain_func
}
```

### 显式加载 vs 隐式加载

```cpp
// ============ 隐式加载（编译时链接）============
// 需要三个文件：
// 1. .h 头文件 - 声明
// 2. .lib 导入库 - 链接时使用
// 3. .dll 动态库 - 运行时使用

// 在代码中直接调用
extern "C" __declspec(dllimport) int dll_function();
void use_implicit() {
    dll_function();  // 系统自动加载 DLL
}

// ============ 显式加载（运行时加载）============
// 不需要 .lib 文件，只需 .dll
void use_explicit() {
    HMODULE hDll = LoadLibrary(TEXT("mylib.dll"));
    if (hDll) {
        typedef int (*FuncPtr)();
        FuncPtr func = (FuncPtr)GetProcAddress(hDll, "dll_function");
        if (func) {
            func();
        }
        FreeLibrary(hDll);
    }
}

// 【面试重点】选择哪种方式？
// - 隐式加载：简单、编译时检查，但启动时必须找到所有 DLL
// - 显式加载：灵活、按需加载、可选功能，但代码复杂
```

### DLL 的加载过程

```
程序启动 → 加载器读取导入表 → 按依赖顺序加载 DLL → 重定位地址 → 执行 DllMain
```

1. Windows 加载器读取 EXE 的导入表
2. 递归加载所有依赖的 DLL
3. 进行地址重定位（如果加载地址与首选地址不同）
4. 调用每个 DLL 的 `DllMain` 函数（`DLL_PROCESS_ATTACH`）

## 1.3 DLL 创建与使用

### Windows 平台 DLL 创建

```cpp
// ============ mylib.h - DLL 导出声明 ============
#pragma once

// 定义导出宏（关键技巧）
#ifdef MYLIB_EXPORTS
    #define MYLIB_API __declspec(dllexport)
#else
    #define MYLIB_API __declspec(dllimport)
#endif

// 导出函数
extern "C" MYLIB_API int add(int a, int b);

// 导出类
class MYLIB_API Calculator {
public:
    static int multiply(int a, int b);
    virtual int divide(int a, int b);
};

// 导出全局变量
extern "C" MYLIB_API int g_version;

// 导出成员函数（注意：类中的函数自动导出）
MYLIB_API void global_func();
```

```cpp
// ============ mylib.cpp - DLL 实现 ============
#define MYLIB_EXPORTS  // 定义导出标志
#include "mylib.h"

// 导出函数实现
extern "C" MYLIB_API int add(int a, int b) {
    return a + b;
}

// 导出类实现
int Calculator::multiply(int a, int b) {
    return a * b;
}

int Calculator::divide(int a, int b) {
    return b != 0 ? a / b : 0;
}

// 导出变量
int g_version = 1;
```

### 使用 .def 文件定义导出（高级方式）

```def
; mylib.def - 模块定义文件
LIBRARY mylib
EXPORTS
    add @1          ; @序号 可用于GetProcAddress
    subtract @2
    Calculator::multiply @3
    ; 私有导出（不放入导出表，但可动态加载）
    private_func PRIVATE
```

```cpp
// ============ 方式二：显式加载 ============
// 只需要：mylib.dll

#include <windows.h>
#include <iostream>

// 定义函数指针类型
typedef int (*AddFunc)(int, int);
typedef int (*MultiplyFunc)(int, int);

void explicit_load_example() {
    // 1. 加载 DLL
    HMODULE hDll = LoadLibrary(TEXT("mylib.dll"));
    if (!hDll) {
        std::cerr << "Failed to load DLL" << std::endl;
        return;
    }

    // 2. 获取函数地址（使用修饰后的名称或 extern "C" 名称）
    AddFunc add = (AddFunc)GetProcAddress(hDll, "add");
    // 或使用序号：GetProcAddress(hDll, MAKEINTRESOURCE(1))

    if (add) {
        int result = add(10, 20);
        std::cout << "10 + 20 = " << result << std::endl;
    }

    // 3. 获取类成员函数比较复杂，通常使用工厂模式
    typedef Calculator* (*CreateCalculatorFunc)();
    CreateCalculatorFunc createCalc = (CreateCalculatorFunc)
        GetProcAddress(hDll, "?createCalculator@@YGPAVCalculator@@XZ");

    // 4. 卸载 DLL
    FreeLibrary(hDll);
}

// 【面试重点】GetProcAddress 的常见问题
// 1. C++ 名称修饰：使用 extern "C" 或 dumpbin /EXPORTS 查看真实名称
// 2. 调用约定：__stdcall、__cdecl、__fastcall 必须匹配
// 3. 序号导出：GetProcAddress(hDll, MAKEINTRESOURCE(1)) 更快更稳定
```

### Qt 跨平台方式

```cpp
// ============ Qt 跨平台动态加载 ============
#include <QLibrary>
#include <QFunctionPointer>
#include <iostream>

// 定义函数指针类型
typedef int (*AddFunc)(int, int);

void qt_load_library() {
    // QLibrary 自动处理平台差异
    QLibrary lib("mylib");  // Windows: mylib.dll, Linux: libmylib.so

    if (lib.load()) {
        // 解析符号（C 风格名称）
        AddFunc add = (AddFunc)lib.resolve("add");

        if (add) {
            std::cout << "10 + 20 = " << add(10, 20) << std::endl;
        } else {
            std::cerr << "Symbol not found: " << lib.errorString().toStdString() << std::endl;
        }

        // 卸载
        lib.unload();
    } else {
        std::cerr << "Failed to load: " << lib.errorString().toStdString() << std::endl;
    }
}

// 【面试重点】Qt 方式的优势
// 1. 跨平台：同一套代码支持 Windows/Linux/macOS
// 2. 自动处理扩展名：.dll、.so、.dylib
// 3. 更好的错误信息
// 4. 支持多个加载实例
```

## 1.4 DLL 常见问题与解决方案

### DLL Hell 问题

```cpp
// ============ 什么是 DLL Hell ============
// 多个程序依赖同一个 DLL 的不同版本时产生的冲突

// 例子：
// 程序 A 需要 lib.dll v1.0
// 程序 B 需要 lib.dll v2.0
// 两个版本只能存在于 System32 目录中，互相覆盖

// ============ 解决方案 ============

// 1. Side-by-Side（WinSxS） assemblies
//    - 每个程序使用自己的 DLL 副本
//    - 放在程序目录而非 System32
//    - 使用 manifest 文件指定版本

// 2. DLL 重定向
//    - 使用 .local 文件或 App-Local DLLs
//    - 程序优先使用自己目录下的 DLL

// 3. 静态链接
//    - 将库代码直接编译到 EXE 中
//    - 缺点：文件变大，无法共享更新
```

### 内存管理问题

```cpp
// ============ 问题：跨 DLL 边界的内存释放 ============
// 【面试重点】常见面试题：为什么会出现崩溃？

// DLL 代码（使用 /MD 或 /MDd 编译）
extern "C" __declspec(dllexport) char* create_buffer() {
    return new char[100];  // 使用 DLL 的堆
}

// EXE 代码（可能使用不同的运行时库）
void bad_example() {
    HMODULE hDll = LoadLibrary(L"mylib.dll");
    auto create = (char*(*)())GetProcAddress(hDll, "create_buffer");

    char* buf = create();  // 在 DLL 的堆上分配
    delete[] buf;          // ❌ 在 EXE 的堆上释放 → 崩溃！
    FreeLibrary(hDll);
}

// ============ 解决方案 ============

// 方案 1：在 DLL 中提供释放函数
extern "C" __declspec(dllexport) void free_buffer(char* p) {
    delete[] p;  // 同一个堆上释放
}

void good_example1() {
    HMODULE hDll = LoadLibrary(L"mylib.dll");
    auto create = (char*(*)())GetProcAddress(hDll, "create_buffer");
    auto destroy = (void(*)(char*))GetProcAddress(hDll, "free_buffer");

    char* buf = create();
    destroy(buf);  // ✅ 在 DLL 中释放
    FreeLibrary(hDll);
}

// 方案 2：使用智能指针 + 自定义删除器
void good_example2() {
    HMODULE hDll = LoadLibrary(L"mylib.dll");
    auto create = (char*(*)())GetProcAddress(hDll, "create_buffer");
    auto destroy = (void(*)(char*))GetProcAddress(hDll, "free_buffer");

    // 使用 lambda 作为删除器
    std::unique_ptr<char[], void(*)(char*)> buf(
        create(),
        [destroy](char* p) { destroy(p); }  // ✅ 自动调用 DLL 的释放函数
    );

    // buf 会自动正确释放
    FreeLibrary(hDll);
}

// 【面试重点】最佳实践
// 1. 确保所有模块使用相同的 CRT 版本（/MD）
// 2. 内存在哪里分配就在哪里释放
// 3. 提供配套的 create/destroy 函数
// 4. 使用智能指针封装 DLL 返回的资源
```

### 异常处理跨 DLL 问题

```cpp
// ============ 问题：异常跨 DLL 边界 ============
// 【面试重点】异常不能跨 DLL 边界抛出（不同的编译器/设置）

// DLL 代码
extern "C" __declspec(dllexport) void risky_function() {
    throw std::runtime_error("Error in DLL");  // 可能不工作
}

// ============ 解决方案：使用错误码 ============
extern "C" __declspec(dllexport) int safe_function(char** error_msg) {
    try {
        // ... 执行操作
        return 0;  // 成功
    } catch (const std::exception& e) {
        if (error_msg) {
            *error_msg = _strdup(e.what());  // 调用者负责释放
        }
        return -1;  // 失败
    }
}

// 或使用 HRESULT（Windows COM 风格）
extern "C" __declspec(dllexport) HRESULT com_style_function() {
    try {
        // ...
        return S_OK;
    } catch (...) {
        return E_FAIL;
    }
}
```

## 1.5 面试重点代码示例

### 完整的 DLL 导出类设计

```cpp
// ============ 接口设计模式（推荐）============
// 使用纯虚接口 + 工厂函数，避免 ABI 问题

// 【面试重点】为什么这样设计？
// 1. 虚函数表保证跨编译器兼容性
// 2. C 风格工厂函数避免名称修饰
// 3. 接口与实现分离

// ============ interface.h - 纯虚接口（不需要编译成 DLL）============
#pragma once

// 定义接口类（不含实现，不导出成员）
class IProcessor {
public:
    virtual ~IProcessor() = default;

    virtual int process(int input) = 0;
    virtual const char* get_name() const = 0;
};

// C 风格工厂函数声明
extern "C" {
    // 创建处理器
    IProcessor* __cdecl create_processor();

    // 销毁处理器
    void __cdecl destroy_processor(IProcessor* p);

    // 获取版本信息
    const char* __cdecl get_processor_version();
}

// ============ processor.cpp - DLL 实现 ============
#include "interface.h"
#include <cstring>

// 实现类
class MyProcessor : public IProcessor {
public:
    int process(int input) override {
        return input * 2;
    }

    const char* get_name() const override {
        return "MyProcessor v1.0";
    }
};

// 工厂函数实现（C 风格，方便GetProcAddress）
extern "C" {
    __declspec(dllexport) IProcessor* __cdecl create_processor() {
        return new MyProcessor();
    }

    __declspec(dllexport) void __cdecl destroy_processor(IProcessor* p) {
        delete p;
    }

    __declspec(dllexport) const char* __cdecl get_processor_version() {
        return "1.0.0";
    }
}
```

### 使用接口的客户端代码

```cpp
// ============ client.cpp - 使用 DLL ============
#include "interface.h"
#include <iostream>
#include <windows.h>

int main() {
    // 加载 DLL
    HMODULE hDll = LoadLibrary(L"processor.dll");
    if (!hDll) {
        std::cerr << "Failed to load DLL" << std::endl;
        return 1;
    }

    // 获取工厂函数
    auto create = (IProcessor*(*)())GetProcAddress(hDll, "create_processor");
    auto destroy = (void(*)(IProcessor*))GetProcAddress(hDll, "destroy_processor");
    auto version = (const char*(*)())GetProcAddress(hDll, "get_processor_version");

    if (create && destroy && version) {
        std::cout << "DLL Version: " << version() << std::endl;

        // 创建处理器
        IProcessor* processor = create();

        // 使用处理器
        int result = processor->process(21);
        std::cout << "Processor name: " << processor->get_name() << std::endl;
        std::cout << "process(21) = " << result << std::endl;

        // 销毁处理器
        destroy(processor);
    }

    FreeLibrary(hDll);
    return 0;
}

// 【面试重点】这种设计的优点
// 1. 编译器无关：虚函数表是标准 ABI
// 2. 版本兼容：新增虚函数不影响旧代码
// 3. 内存安全：成对的 create/destroy
// 4. 可选功能：可以用 GetProcAddress 判断函数是否存在
```

### 函数指针的高级用法

```cpp
// ============ 动态调用表 ============
#include <windows.h>
#include <functional>
#include <map>
#include <string>

class DynamicLoader {
public:
    DynamicLoader(const wchar_t* dll_name) : m_handle(nullptr) {
        m_handle = LoadLibrary(dll_name);
    }

    ~DynamicLoader() {
        if (m_handle) {
            FreeLibrary(m_handle);
        }
    }

    // 获取函数模板
    template<typename FuncType>
    std::function<FuncType> get(const std::string& name) {
        if (!m_handle) return nullptr;

        auto proc = GetProcAddress(m_handle, name.c_str());
        return reinterpret_cast<FuncType*>(proc);
    }

    bool is_loaded() const { return m_handle != nullptr; }

private:
    HMODULE m_handle;
};

// 使用示例
void advanced_usage() {
    DynamicLoader loader(L"user32.dll");

    if (loader.is_loaded()) {
        // 动态获取 MessageBox
        auto msgBox = loader.get<int(HWND, LPCSTR, LPCSTR, UINT)>("MessageBoxA");

        if (msgBox) {
            msgBox(nullptr, "Hello from dynamic loader!", "DLL Test", MB_OK);
        }
    }
}
```

## 1.6 DLL 面试题总结

### 常见面试问题

1. **静态库和动态库的区别是什么？**
   - 链接时机、代码共享、更新方式、内存占用

2. **什么是 DLL Hell？如何解决？**
   - 版本冲突问题，使用 Side-by-Side、App-Local DLLs

3. **extern "C" 的作用是什么？**
   - 禁用 C++ 名称修饰，使用 C 链接约定

4. **为什么不能用 delete 释放 DLL 中 new 的内存？**
   - 不同的堆（Heap），需要同一运行时库

5. **__declspec(dllexport) 和 __declspec(dllimport) 的区别？**
   - dllexport：导出符号（DLL 中使用）
   - dllimport：导入符号（使用 DLL 时使用）

6. **如何查看 DLL 的导出函数？**
   - dumpbin /EXPORTS、Dependencies.exe、Dependency Walker

7. **LoadLibrary 和隐式链接的区别？**
   - 加载时机、灵活性、错误处理

## 2.1 插件系统架构

### 插件系统的设计目标

```
┌─────────────────────────────────────────────────────────────┐
│                        主程序 (Host)                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │  插件管理器   │  │   核心功能    │  │   UI 界面     │      │
│  └──────┬───────┘  └──────────────┘  └──────────────┘      │
│         │                                                    │
│         │ 加载/卸载/调用                                      │
└─────────┼────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────────┐
│                      插件接口 (Interface)                     │
│  - 定义统一的契约                                             │
│  - 版本控制                                                   │
│  - 元数据描述                                                 │
└─────────────────────────────────────────────────────────────┘
          │
    ┌─────┴─────┬─────────────┬─────────────┐
    ▼           ▼             ▼             ▼
┌────────┐ ┌────────┐   ┌────────┐   ┌────────┐
│ 插件 A │ │ 插件 B │   │ 插件 C │   │ 插件 D │
│(DLL)   │ │(DLL)   │   │(DLL)   │   │(DLL)   │
└────────┘ └────────┘   └────────┘   └────────┘
```

### 插件系统的核心组件

```cpp
// ============ 插件系统的核心组件 ============

// 1. 插件接口 (Plugin Interface)
//    - 定义插件必须实现的方法
//    - 作为主程序和插件之间的契约

// 2. 插件加载器 (Plugin Loader)
//    - 负责扫描、加载插件
//    - 管理插件生命周期

// 3. 插件管理器 (Plugin Manager)
//    - 维护插件注册表
//    - 处理插件依赖关系
//    - 提供插件查询/调用接口

// 4. 插件描述符 (Plugin Descriptor)
```

```cpp
#include <QtCore/QDebug>

PluginManager::PluginManager(QObject* parent) : QObject(parent) {}

void PluginManager::loadPlugins(const QString& pluginDir) {
    QDir dir(pluginDir);

    // 查找所有 DLL/SO 文件
    foreach (const QString& fileName, dir.entryList(QDir::Files)) {
        QPluginLoader* loader = new QPluginLoader(dir.absoluteFilePath(fileName), this);

        // 加载插件
        if (loader->load()) {
            QObject* plugin = loader->instance();

            // 检查插件是否实现了我们的接口
            ImageFilterInterface* filter = qobject_cast<ImageFilterInterface*>(plugin);

            if (filter) {
                filters_.append(filter);
                loaders_.append(loader);

                qDebug() << "Loaded plugin:" << filter->filterName()
                         << "v" << filter->filterVersion();

                emit pluginLoaded(filter->filterName());
            } else {
                qWarning() << "Plugin does not implement ImageFilterInterface:"
                           << fileName;
                loader->unload();
                delete loader;
            }
        } else {
            qWarning() << "Failed to load plugin:" << fileName
                       << "Error:" << loader->errorString();
            emit pluginLoadFailed(fileName, loader->errorString());
            delete loader;
        }
    }
}

ImageFilterInterface* PluginManager::findFilter(const QString& name) const {
    foreach (ImageFilterInterface* filter, filters_) {
        if (filter->filterName().compare(name, Qt::CaseInsensitive) == 0) {
            return filter;
        }
    }
    return nullptr;
}
```

```cpp
// ============ pluginloader.cpp ============
#include "pluginloader.h"
#include <stdexcept>
#include <iostream>

PluginLoader::~PluginLoader() {
    // 卸载所有插件
    std::lock_guard<std::mutex> lock(mutex_);
    for (auto& [name, data] : plugins_) {
        unloadPluginData(data);
    }
    plugins_.clear();
}

bool PluginLoader::load(const std::string& pluginPath) {
    std::lock_guard<std::mutex> lock(mutex_);

    PluginData data{};
    if (!loadPluginData(pluginPath, data)) {
        return false;
    }

    // 先获取插件信息
    if (data.getInfoFunc) {
        const PluginInfo* info = data.getInfoFunc();
        if (info) {
            std::string name = info->name;
            plugins_[name] = std::move(data);
            std::cout << "Loaded plugin: " << name
                      << " v" << info->version.toString() << std::endl;
            return true;
        }
    }

    unloadPluginData(data);
    return false;
}

bool PluginLoader::unload(const std::string& pluginName) {
    std::lock_guard<std::mutex> lock(mutex_);

    auto it = plugins_.find(pluginName);
    if (it == plugins_.end()) {
        return false;
    }

    unloadPluginData(it->second);
    plugins_.erase(it);
    return true;
}

IPlugin* PluginLoader::getPlugin(const std::string& name) const {
    std::lock_guard<std::mutex> lock(mutex_);

    auto it = plugins_.find(name);
    if (it != plugins_.end() && it->second.instance) {
        return it->second.instance;
    }
    return nullptr;
}

std::vector<IPlugin*> PluginLoader::getAllPlugins() const {
    std::lock_guard<std::mutex> lock(mutex_);

    std::vector<IPlugin*> result;
    for (const auto& [name, data] : plugins_) {
        if (data.instance) {
            result.push_back(data.instance);
        }
    }
    return result;
}

bool PluginLoader::getPluginInfo(const std::string& pluginPath, PluginInfo& info) const {
    PluginData data{};
    if (!loadPluginData(pluginPath, data)) {
        return false;
    }

    if (data.getInfoFunc) {
        const PluginInfo* pluginInfo = data.getInfoFunc();
        if (pluginInfo) {
            info = *pluginInfo;
            unloadPluginData(data);
            return true;
        }
    }

    unloadPluginData(data);
    return false;
}

bool PluginLoader::loadPluginData(const std::string& path, PluginData& data) {
#ifdef _WIN32
    data.handle = LoadLibraryA(path.c_str());
    if (!data.handle) {
        std::cerr << "Failed to load DLL: " << path << std::endl;
        return false;
    }

    // 获取函数地址
    data.getInfoFunc = (GetPluginInfoFunc)GetProcAddress(data.handle, "getPluginInfo");
    data.createFunc = (CreatePluginFunc)GetProcAddress(data.handle, "createPlugin");
    data.destroyFunc = (DestroyPluginFunc)GetProcAddress(data.handle, "destroyPlugin");

    if (!data.createFunc || !data.destroyFunc) {
        std::cerr << "Plugin missing required functions" << std::endl;
        FreeLibrary(data.handle);
        return false;
    }

    // 创建插件实例
    data.instance = data.createFunc();
    if (!data.instance) {
        std::cerr << "Failed to create plugin instance" << std::endl;
        FreeLibrary(data.handle);
        return false;
    }

    // 初始化插件
    if (!data.instance->initialize()) {
        std::cerr << "Plugin initialization failed" << std::endl;
        data.destroyFunc(data.instance);
        FreeLibrary(data.handle);
        return false;
    }

#else // Linux/macOS
    data.handle = dlopen(path.c_str(), RTLD_LAZY);
    if (!data.handle) {
        std::cerr << "Failed to load library: " << dlerror() << std::endl;
        return false;
    }

    // 获取函数地址
    data.getInfoFunc = (GetPluginInfoFunc)dlsym(data.handle, "getPluginInfo");
    data.createFunc = (CreatePluginFunc)dlsym(data.handle, "createPlugin");
    data.destroyFunc = (DestroyPluginFunc)dlsym(data.handle, "destroyPlugin");

    if (!data.createFunc || !data.destroyFunc) {
        std::cerr << "Plugin missing required functions" << std::endl;
        dlclose(data.handle);
        return false;
    }

    // 创建插件实例
    data.instance = data.createFunc();
    if (!data.instance || !data.instance->initialize()) {
        if (data.instance && data.destroyFunc) {
            data.destroyFunc(data.instance);
        }
        dlclose(data.handle);
        return false;
    }
#endif

    return true;
}

void PluginLoader::unloadPluginData(PluginData& data) {
    if (data.instance && data.destroyFunc) {
        data.instance->shutdown();
        data.destroyFunc(data.instance);
        data.instance = nullptr;
    }

#ifdef _WIN32
    if (data.handle) {
        FreeLibrary(data.handle);
        data.handle = nullptr;
    }
#else
    if (data.handle) {
        dlclose(data.handle);
        data.handle = nullptr;
    }
#endif
}
```

```cpp
public:
```

```cpp
    void fetch() {}  // 非虚函数，不在 vtable 中

// 【面试重点】vtable 布局
//
// Animal 的 vtable（纯虚函数）:
// +------------------+  RTTI 信息
// | &Animal::speak    |  (可能设为 __cxa_pure_virtual)
// +------------------+
// | &Animal::eat      |
// +------------------+
// | &Animal::~Animal  |
// +------------------+
//
// Dog 的 vtable:
// +------------------+  RTTI 信息 (typeinfo for Dog)
// | &Dog::speak       |  (覆盖了 Animal::speak)
// +------------------+
// | &Dog::eat         |  (覆盖了 Animal::eat)
// +------------------+
// | &Dog::~Dog        |
// +------------------+

// 【验证代码】打印虚函数表
void print_vtable(void* obj) {
    std::cout << "Object at: " << obj << std::endl;

    // vptr 在对象起始位置
    void** vptr = *(void***)obj;

    std::cout << "vtable at: " << vptr << std::endl;

    // 打印前几个虚函数地址
    for (int i = 0; i < 3; ++i) {
        std::cout << "  vtable[" << i << "] = " << vptr[i] << std::endl;
    }
}

void test_vtable() {
    Dog dog;
    Animal* animal = &dog;

    print_vtable(&dog);
    print_vtable(animal);  // 相同的 vtable
}
```

```cpp
// ============ 示例：多重继承的 vtable ============

class Base1 {
public:
    virtual void f1() {}
    virtual void f2() {}
    int b1;
};

class Base2 {
public:
    virtual void g1() {}
    virtual void g2() {}
    int b2;
};

class Multi : public Base1, public Base2 {
public:
    void f1() override {}
    void f2() override {}
    void g1() override {}
    void g2() override {}
    int m;
};

// 【面试重点】多重继承的 vtable
//
// Multi 有两个 vtable：
//
// vtable1 (for Base1 subobject):
// +---------------+
// | &Multi::f1    |
// +---------------+
// | &Multi::f2    |
// +---------------+
// | RTTI (Multi)  |
// +---------------+
//
// vtable2 (for Base2 subobject):
// +---------------+
// | &Multi::g1    |  (注意：调用时需要 this 指针调整)
// +---------------+
// | &Multi::g2    |
// +---------------+
// | RTTI (Multi)  |
// +---------------+
// | offset to top |  (调整到 Multi* 的偏移)
// +---------------+

// 【关键点】虚函数调用时的 this 调整
void test_multi_vtable() {
    Multi m;

    Base1* b1 = &m;
    Base2* b2 = &m;

    b1->f1();  // 直接调用，不需要 this 调整
    b2->g1();  // 间接调用，需要 this 调整（减去 sizeof(Base1)）

    // 编译器生成的代码类似：
    // b1->f1():
    //   vptr = *b1
    //   func = vptr[0]
    //   func(b1)  // this 指针不变
    //
    // b2->g1():
    //   vptr = *b2
    //   thunk = vptr[0]  // 可能是 thunk 代码
    //   thunk(b2 - offset)  // this 指针调整
}
```

```cpp
// ============ 虚函数调用的底层实现 ============

class Base {
public:
    virtual void func() {}
};

class Derived : public Base {
public:
    void func() override {}
};

void call_virtual(Base* obj) {
    obj->func();
}

// 【面试重点】编译器生成的类似代码（伪汇编）:
//
// call_virtual:
//     mov     rax, [rdi]      ; 加载 vptr
//     call    [rax + 0]       ; 调用 vtable[0]
//     ret
//
// 对比非虚函数调用：
//
// call_nonvirtual(Base* obj):
//     call    Base::func      ; 直接调用，地址固定
//     ret

// 【性能对比】
// - 虚函数调用：2 次内存访问（vptr + vtable entry）
// - 非虚函数：直接跳转，可内联
// - 性能差异：现代 CPU 有分支预测，差异不大

// 【优化技巧】final 关键字
class FinalDerived final : public Base {
public:
    void func() override final {}  // final 表示不可再覆盖
};

// 编译器可以优化：
// FinalDerived* obj;
// obj->func();  // 可能被内联，因为编译器知道没有进一步覆盖
```

## 3.5 对象模型面试题总结

### 常见面试问题

1. **什么是 vtable？它存储在哪里？**
   - 每个有虚函数的类有一个 vtable，存储在只读数据区
   - 每个对象有一个 vptr，指向类的 vtable

2. **多重继承下，this 指针如何调整？**
   - 将基类指针转换为派生类指针时，编译器插入偏移量调整

3. **什么是对象切片？**
   - 派生类对象按值赋给基类对象时，派生类部分被"切掉"

4. **dynamic_cast 的工作原理？**
   - 使用 RTTI 遍历继承树，找到目标类型并调整指针

5. **虚继承的开销是什么？**
   - 额外的虚基类表指针、间接访问开销、对象大小增加

6. **为什么析构函数应该是虚函数？**
   - 确保通过基类指针删除派生类对象时，调用正确的析构函数

## 4.2 Rule of Three / Rule of Five / Rule of Zero

```cpp
// ============ Rule of Three（三法则）============
// 如果你需要定义以下任何一个，你应该定义全部三个：
// 1. 析构函数
// 2. 复制构造函数
// 3. 复制赋值运算符

class RuleOfThree {
public:
    // 【面试重点】需要手动管理资源时，遵循 Rule of Three
    RuleOfThree(const char* str) {
        size_ = strlen(str) + 1;
        data_ = new char[size_];
        strcpy(data_, str);
    }

    // 1. 析构函数：释放资源
    ~RuleOfThree() {
        delete[] data_;
    }

    // 2. 复制构造函数：深拷贝
    RuleOfThree(const RuleOfThree& other) {
        size_ = other.size_;
        data_ = new char[size_];
        strcpy(data_, other.data_);
    }

    // 3. 复制赋值运算符：深拷贝 + 自赋值检查
    RuleOfThree& operator=(const RuleOfThree& other) {
        if (this != &other) {  // 自赋值检查！
            delete[] data_;     // 释放旧资源
            size_ = other.size_;
            data_ = new char[size_];
            strcpy(data_, other.data_);
        }
        return *this;
    }

private:
    char* data_;
    size_t size_;
};

// ============ Rule of Five（五法则，C++11）============
// 在 Rule of Three 基础上，还需要定义：
// 4. 移动构造函数
// 5. 移动赋值运算符

class RuleOfFive {
public:
    RuleOfFive(const char* str) {
        size_ = strlen(str) + 1;
        data_ = new char[size_];
        strcpy(data_, str);
    }

    ~RuleOfFive() {
        delete[] data_;
    }

    // 复制构造（深拷贝）
    RuleOfFive(const RuleOfFive& other) {
        size_ = other.size_;
        data_ = new char[size_];
        strcpy(data_, other.data_);
    }

    // 复制赋值
    RuleOfFive& operator=(const RuleOfFive& other) {
        if (this != &other) {
            delete[] data_;
            size_ = other.size_;
            data_ = new char[size_];
            strcpy(data_, other.data_);
        }
        return *this;
    }

    // 【面试重点】4. 移动构造（窃取资源）
    RuleOfFive(RuleOfFive&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;  // 置空，防止析构时释放
        other.size_ = 0;
    }

    // 【面试重点】5. 移动赋值
    RuleOfFive& operator=(RuleOfFive&& other) noexcept {
        if (this != &other) {
            delete[] data_;      // 释放自己的资源
            data_ = other.data_;  // 窃取
            size_ = other.size_;
            other.data_ = nullptr;
            other.size_ = 0;
        }
        return *this;
    }

private:
    char* data_;
    size_t size_;
};

// ============ Rule of Zero（零法则，C++11 最佳实践）============
// 【面试重点】尽可能让编译器自动生成特殊成员函数
// 使用标准库容器和智能指针管理资源

class RuleOfZero {
public:
    RuleOfZero(const std::string& str) : data_(str) {}

    // 不需要声明任何特殊成员函数！
    // 编译器自动生成的都是正确的：
    // - 默认构造函数
    // - 复制构造（std::string 的复制）
    // - 复制赋值（std::string 的赋值）
    // - 移动构造（std::string 的移动）
    // - 移动赋值（std::string 的移动）
    // - 析构函数（std::string 的析构）

private:
    std::string data_;  // 管理自己的资源
};

// 【何时使用哪个法则？】
// - Rule of Zero：优先使用，使用智能指针和标准容器
// - Rule of Five：需要管理原始资源时（文件、套接字、互斥锁等）
// - Rule of Three：老代码或不支持移动语义时
```

### RAII 与资源管理

```cpp
// ============ RAII (Resource Acquisition Is Initialization) ============
// 【面试重点】RAII 是 C++ 资源管理的核心原则

// 不使用 RAII（错误方式）
void bad_file_handling() {
    FILE* f = fopen("file.txt", "w");
    if (!f) return;

    // ... 使用文件 ...

    if (some_error) {
        fclose(f);  // 需要记得手动关闭
        return;
    }

    fclose(f);  // 多个返回路径都要关闭
}

// 使用 RAII（正确方式）
class FileHandle {
public:
    FileHandle(const char* filename, const char* mode) {
        file_ = fopen(filename, mode);
    }

    ~FileHandle() {
        if (file_) {
            fclose(file_);  // 自动关闭
        }
    }

    // 禁止复制
    FileHandle(const FileHandle&) = delete;
    FileHandle& operator=(const FileHandle&) = delete;

    // 允许移动
    FileHandle(FileHandle&& other) noexcept : file_(other.file_) {
        other.file_ = nullptr;
    }

    FILE* get() { return file_; }

private:
    FILE* file_;
};

void good_file_handling() {
    FileHandle f("file.txt", "w");
    if (!f.get()) return;

    // ... 使用文件 ...
    // 函数结束时自动关闭，无论是否抛异常
}

// 【RAII 的优势】
// 1. 异常安全：异常发生时资源正确释放
// 2. 防止泄漏：不需要记住释放资源
// 3. 代码简洁：减少样板代码

// 【标准库的 RAII 类】
// - std::unique_ptr / std::shared_ptr：管理动态内存
// - std::lock_guard / std::unique_lock：管理互斥锁
// - std::fstream：管理文件句柄
// - std::vector / std::string：管理容器内存
```

```cpp
        data_ = new int[size];
        size_ = size;
    }

    // 编译器生成的复制构造函数是浅拷贝！
    // ~ShallowCopy() { delete[] data_; }  // 双重删除！

    int* data_;
    size_t size_;

    ShallowCopy b = a;  // 浅拷贝！b.data_ == a.data_
    // a 和 b 析构时会删除同一块内存两次！崩溃！

class DeepCopy {
public:
    DeepCopy(int size) {
        data_ = new int[size];
        size_ = size;
    }

    // 自定义复制构造：深拷贝
    DeepCopy(const DeepCopy& other) {
        size_ = other.size_;
        data_ = new int[size_];
        std::copy(other.data_, other.data_ + size_, data_);
    }

    ~DeepCopy() {
        delete[] data_;
    }

    DeepCopy& operator=(const DeepCopy& other) {
        if (this != &other) {
            delete[] data_;
            size_ = other.size_;
            data_ = new int[size_];
            std::copy(other.data_, other.data_ + size_, data_);
        }
        return *this;
    }

private:
    int* data_;
    size_t size_;
};
```

```cpp
public:

    ~MoveExample() {

        std::cout << "Resource freed" << std::endl;

    int* data_;
    size_t size_;
};
```

```cpp
// T& &&  → T&
// T&& &  → T&
```

## 4.6 复制交换（Copy-and-Swap）惯用法

```cpp
// ============ Copy-and-Swap 惯用法 ============
// 【面试重点】一种优雅实现赋值运算符的方法

class CopyAndSwap {
public:
    CopyAndSwap(size_t size) : size_(size), data_(new int[size]) {}

    // 析构函数
    ~CopyAndSwap() {
        delete[] data_;
    }

    // 复制构造函数
    CopyAndSwap(const CopyAndSwap& other)
        : size_(other.size_), data_(new int[other.size_]) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // 【重点】交换函数
    friend void swap(CopyAndSwap& first, CopyAndSwap& second) noexcept {
        using std::swap;
        swap(first.size_, second.size_);
        swap(first.data_, second.data_);
    }

    // 【重点】复制赋值运算符：使用 Copy-and-Swap
    CopyAndSwap& operator=(CopyAndSwap other) {  // 注意：按值传递！
        swap(*this, other);  // 交换 this 和 other
        return *this;        // other 析构时自动释放旧资源
    }

    // 移动构造函数
    CopyAndSwap(CopyAndSwap&& other) noexcept
        : size_(0), data_(nullptr) {
        swap(*this, other);
    }

private:
    int* data_;
    size_t size_;
};

// 【Copy-and-Swap 的优势】
void test_copy_and_swap() {
    CopyAndSwap a1(100);
    CopyAndSwap a2(200);

    a1 = a2;  // 赋值
}

// 执行过程：
// 1. a2 通过复制构造函数传递给 operator=（创建临时对象 other）
// 2. swap 交换 *this 和 other
// 3. other（原来的 a1）在函数结束时析构，释放旧资源

// 【优势】
// 1. 自动处理自赋值：a = a 也能正确工作
// 2. 异常安全：复制失败不影响原对象
// 3. 代码简洁：只有一个 swap 函数
// 4. 自动获得移动赋值：当参数是右值时

// 【劣势】
// 1. 总是需要一次复制（即使是移动赋值）
// 2. 对于大型对象可能有性能问题

// 【优化版本】分别提供复制和移动赋值
CopyAndSwap& operator=(const CopyAndSwap& other) {  // 复制赋值
    if (this != &other) {
        CopyAndSwap temp(other);
        swap(*this, temp);
    }
    return *this;
}

CopyAndSwap& operator=(CopyAndSwap&& other) noexcept {  // 移动赋值
    swap(*this, other);
    return *this;
}
```

## 4.7 完整的资源管理类示例

```cpp
// ============ 完整的 Rule of Five 实现 ============

class Buffer {
public:
    // ========== 1. 默认构造函数 ==========
    Buffer() : data_(nullptr), size_(0) {}

    // ========== 2. 带参构造函数 ==========
    explicit Buffer(size_t size) : size_(size) {
        if (size > 0) {
            data_ = new char[size];
        }
    }

    // ========== 3. 复制构造函数（深拷贝）==========
    Buffer(const Buffer& other) : size_(other.size_) {
        if (other.size_ > 0) {
            data_ = new char[other.size_];
            std::copy(other.data_, other.data_ + size_, data_);
        }
    }

    // ========== 4. 复制赋值运算符 ==========
    Buffer& operator=(const Buffer& other) {
        if (this != &other) {
            // 创建临时副本（异常安全）
            char* new_data = nullptr;
            size_t new_size = other.size_;

            if (new_size > 0) {
                new_data = new char[new_size];
                std::copy(other.data_, other.data_ + new_size, new_data);
            }

            // 成功后才修改当前对象
            delete[] data_;
            data_ = new_data;
            size_ = new_size;
        }
        return *this;
    }

    // ========== 5. 移动构造函数 ==========
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;
        other.size_ = 0;
    }

    // ========== 6. 移动赋值运算符 ==========
    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] data_;
            data_ = other.data_;
            size_ = other.size_;
            other.data_ = nullptr;
            other.size_ = 0;
        }
        return *this;
    }

    // ========== 7. 析构函数 ==========
    ~Buffer() {
        delete[] data_;
    }

    // ========== 其他成员函数 ==========
    size_t size() const { return size_; }
    char* data() { return data_; }
    const char* data() const { return data_; }

    bool empty() const { return size_ == 0; }

private:
    char* data_;
    size_t size_;
};

// 使用示例
void test_buffer() {
    // 默认构造
    Buffer b1;

    // 带参构造
    Buffer b2(1024);

    // 复制
    Buffer b3 = b2;  // 复制构造
    Buffer b4(512);
    b4 = b2;         // 复制赋值

    // 移动
    Buffer b5 = std::move(b2);  // 移动构造
    b1 = std::move(b3);         // 移动赋值

    // b2 和 b3 现在是空的，但仍然是有效状态
}

// 【面试重点】检查一个类是否正确实现了资源管理
// 1. 析构函数是否释放资源？
// 2. 复制构造是否深拷贝？
// 3. 赋值运算符是否处理自赋值？
// 4. 移动操作是否将源对象置为有效状态？
// 5. 是否异常安全？
```

## 4.8 构造析构面试题总结

### 常见面试问题

1. **什么是 Rule of Three？**
   - 如果你需要自定义析构函数、复制构造或复制赋值中的任何一个，你应该自定义全部三个

2. **为什么应该优先使用 Rule of Zero？**
   - 使用智能指针和标准容器管理资源，让编译器生成正确的特殊成员函数

3. **初始化列表和构造函数体内赋值的区别？**
   - 初始化列表直接初始化成员
   - 构造函数体内赋值是先默认构造再赋值
   - const 成员、引用成员必须使用初始化列表

4. **为什么要用 explicit？**
   - 防止意外的隐式类型转换
   - 让代码意图更明确

5. **虚析构函数的作用？**
   - 确保通过基类指针删除派生类对象时，调用正确的析构函数
   - 防止资源泄漏

6. **std::move 和 std::forward 的区别？**
   - std::move：无条件转换为右值引用，用于移动
   - std::forward：根据模板参数保持值类别，用于转发

7. **什么是 RAII？**
   - 资源获取即初始化，利用对象生命周期管理资源

8. **Copy-and-Swap 惯用法的优势？**
   - 自动处理自赋值
   - 异常安全
   - 代码简洁

```cpp
    // - 信号没有返回值
```

```cpp
// 它通过索引动态调用方法
int MyClass::qt_metacall(QMetaObject::Call _c, int _id, void** _a) {
    _id = QObject::qt_metacall(_c, _id, _a);
    if (_id < 0)
        return _id;

    // 根据索引分发到具体的信号/槽
    switch (_id) {
    case 0: valueChanged((*reinterpret_cast< int(*)>(_a[1]))); break;
    case 1: errorOccurred((*reinterpret_cast< QString(*)>(_a[1]))); break;
    case 2: onValueChanged((*reinterpret_cast< int(*)>(_a[1]))); break;
    case 3: onError((*reinterpret_cast< QString(*)>(_a[1]))); break;
    default: ;
    }
    return _id - 3;
}
```

## 5.2 信号槽连接类型详解

### 五种连接类型

```cpp
// ============ Qt::ConnectionType 完整示例 ============

#include <QObject>
#include <QThread>
#include <QDebug>

class Sender : public QObject {
    Q_OBJECT
public:
    Sender(QObject* parent = nullptr) : QObject(parent) {}

    void emitSignal() {
        qDebug() << "Sender: emitting signal from thread:" << QThread::currentThread();
        emit testSignal();
    }

signals:
    void testSignal();
};

class Receiver : public QObject {
    Q_OBJECT
public:
    Receiver(QObject* parent = nullptr) : QObject(parent) {}

public slots:
    void onReceive() {
        qDebug() << "Receiver: slot called in thread:" << QThread::currentThread();
    }
};

// ========== 1. AutoConnection（默认）==========
// 行为：自动判断
// - 同线程：相当于 DirectConnection
// - 跨线程：相当于 QueuedConnection

void test_auto_connection() {
    Sender sender;
    Receiver receiver;

    // 默认连接类型
    QObject::connect(&sender, &Sender::testSignal,
                     &receiver, &Receiver::onReceive);
    // 等价于 Qt::AutoConnection

    sender.emitSignal();
    // 输出：两个都在主线程，同步调用
}

// ========== 2. DirectConnection（直接连接）==========
// 行为：槽函数在信号发射线程立即执行（同步）
// 用途：性能优化、确保调用顺序
// 风险：跨线程调用可能导致线程安全问题

void test_direct_connection() {
    Sender sender;
    Receiver receiver;

    QObject::connect(&sender, &Sender::testSignal,
                     &receiver, &Receiver::onReceive,
                     Qt::DirectConnection);

    sender.emitSignal();
    // 同步调用，槽在发送者线程执行
}

// ========== 3. QueuedConnection（队列连接）==========
// 行为：槽函数在接收者线程的事件循环中执行（异步）
// 用途：跨线程通信、避免阻塞
// 优点：线程安全

void test_queued_connection() {
    QThread workerThread;
    workerThread.start();

    Sender sender;
    Receiver* receiver = new Receiver();
    receiver->moveToThread(&workerThread);  // 移动到工作线程

    QObject::connect(&sender, &Sender::testSignal,
                     receiver, &Receiver::onReceive,
                     Qt::QueuedConnection);

    sender.emitSignal();
    // 输出：
    // Sender: emitting signal from thread: QThread(主线程)
    // Receiver: slot called in thread: QThread(工作线程)

    workerThread.quit();
    workerThread.wait();
    delete receiver;
}

// ========== 4. BlockingQueuedConnection（阻塞队列）==========
// 行为：信号发射线程会阻塞等待槽函数执行完成
// 限制：只能用于跨线程，同线程会死锁！
// 用途：需要同步等待跨线程调用完成

void test_blocking_queued_connection() {
    QThread workerThread;
    workerThread.start();

    Sender sender;
    Receiver* receiver = new Receiver();
    receiver->moveToThread(&workerThread);

    QObject::connect(&sender, &Sender::testSignal,
                     receiver, &Receiver::onReceive,
                     Qt::BlockingQueuedConnection);

    sender.emitSignal();  // 会阻塞等待槽函数执行完成

    workerThread.quit();
    workerThread.wait();
    delete receiver;
}

// 【面试重点】死锁风险！
// ❌ 错误：同线程使用 BlockingQueuedConnection
void dangerous_usage() {
    Sender sender;
    Receiver receiver;

    QObject::connect(&sender, &Sender::testSignal,
                     &receiver, &Receiver::onReceive,
                     Qt::BlockingQueuedConnection);  // 危险！

    sender.emitSignal();  // 死锁！
    // 原因：同线程的事件循环被阻塞，无法处理队列中的事件
}

// ========== 5. UniqueConnection（唯一连接）==========
// 行为：防止重复连接
// 用法：与其他连接类型组合使用

void test_unique_connection() {
    Sender sender;
    Receiver receiver;

    // 第一次连接：成功
    QObject::connect(&sender, &Sender::testSignal,
                     &receiver, &Receiver::onReceive,
                     Qt::UniqueConnection);

    // 第二次连接：失败（重复连接）
    bool connected = QObject::connect(&sender, &Sender::testSignal,
                                      &receiver, &Receiver::onReceive,
                                      Qt::UniqueConnection);

    qDebug() << "Second connection result:" << connected;  // false
}
```

### 连接类型对比表

| 连接类型 | 同线程行为 | 跨线程行为 | 使用场景 | 注意事项 |
|---------|-----------|-----------|---------|---------|
| **AutoConnection** | Direct | Queued | 默认选项 | 自动选择最安全的方式 |
| **DirectConnection** | 立即同步调用 | 在发送者线程调用 | 性能优化 | 跨线程时要注意线程安全 |
| **QueuedConnection** | 事件队列 | 事件队列 | 跨线程通信 | 线程安全，但异步 |
| **BlockingQueued** | 死锁！ | 阻塞等待 | 需要同步等待 | 只能跨线程使用 |
| **UniqueConnection** | - | - | 防止重复 | 需要与其他类型组合 |

## 5.5 信号槽面试重点

### 常见面试问题

**1. emit 是关键字吗？**
```cpp
// 【面试重点】emit 不是 C++ 关键字！
// 它只是一个空的宏定义
#define emit

// 以下两种写法完全等价：
emit mySignal(value);
mySignal(value);

// emit 的作用只是提高代码可读性
```

**2. 信号槽是线程安全的吗？**
```
【详细解释】
- DirectConnection：不是线程安全的，槽在发送者线程执行
- QueuedConnection：是线程安全的，Qt 通过事件队列实现
- 跨线程 AutoConnection：自动使用 QueuedConnection，线程安全
```

**3. 跨线程信号槽如何保证安全？**
```
Qt 的实现机制：
1. 参数被复制（通过 QMetaType）
2. 创建 QMetaCallEvent 包含参数
3. 投递到接收者线程的事件队列
4. 接收者线程处理事件时调用槽函数
```

**4. 信号槽与普通函数调用的区别？**
```
                    信号槽                    普通函数调用
调用方式            间接调用（元对象）         直接调用
耦合程度            松耦合                     紧耦合
跨线程支持          内置支持                   需要手动实现
性能开销            较高                       低
```

**5. 信号槽的性能开销？**
```
- DirectConnection：开销接近虚函数调用，比直接函数调用慢约 2-3 倍
- QueuedConnection：需要复制参数、创建事件、投递队列，开销较大
- 建议：频繁调用的场景考虑直接函数调用
```

```cpp
// 【面试重点】方式一的问题
// 1. QThread 对象本身在主线程，run() 在新线程
// 2. 容易混淆：槽函数在哪个线程执行？
// 3. 不符合 Qt 的设计理念

// ==========

// 【方式二】使用 moveToThread（推荐）==========
class Worker : public QObject {
    Q_OBJECT
public:
    Worker(QObject* parent = nullptr) : QObject(parent) {}

public slots:
    void doWork() {
        // 【重点】这段代码在 moveToThread 指定的线程中执行
        for (int i = 0; i < 10; ++i) {
            qDebug() << "Working..." << i << "Thread:" << QThread::currentThread();
            QThread::msleep(100);
        }
        emit workFinished();
    }

signals:
    void workFinished();
};

void test_worker() {
    QThread workerThread;
    Worker worker;

    // 【关键】将 Worker 对象移动到工作线程
    worker.moveToThread(&workerThread);

    // 连接信号槽
    QObject::connect(&workerThread, &QThread::started,
                     &worker, &Worker::doWork);
    QObject::connect(&worker, &Worker::workFinished,
                     &workerThread, &QThread::quit);
    QObject::connect(&workerThread, &QThread::finished,
                     &workerThread, &QThread::deleteLater);

    workerThread.start();
    workerThread.wait();
}

// 【面试重点】两种方式的区别
// | 特性 | 方式一（重写run） | 方式二（moveToThread） |
// |------|------------------|----------------------|
// | 对象所在线程 | 主线程 | 工作线程 |
// | 代码位置 | run() 方法 | 槽函数 |
// | 灵活性 | 低 | 高（可以有多个槽）|
// | 推荐度 | 不推荐 | 推荐 |
```

```cpp
// ============ QMetaObject::invokeMethod ============
// 【面试重点】另一种跨线程调用方式

class InvokeMethodExample : public QObject {
    Q_OBJECT
public:
    InvokeMethodExample(QObject* parent = nullptr) : QObject(parent) {}

    void crossThreadCall() {
        QThread* thread = new QThread;
        RemoteObject* obj = new RemoteObject;
        obj->moveToThread(thread);

        thread->start();

        // 【重点】在另一个线程中调用方法
        // Qt::QueuedConnection 确保在目标线程执行
        QMetaObject::invokeMethod(obj, "doWork",
                                  Qt::QueuedConnection,
                                  Q_ARG(int, 42));

        // 【重点】阻塞等待调用完成
        QMetaObject::invokeMethod(obj, "getResult",
                                  Qt::BlockingQueuedConnection,
                                  Q_RETURN_ARG(int, result));

        qDebug() << "Result:" << result;
    }

private:
    int result;
};

class RemoteObject : public QObject {
    Q_OBJECT
public:
    RemoteObject(QObject* parent = nullptr) : QObject(parent) {}

public slots:
    void doWork(int value) {
        qDebug() << "Working in thread:" << QThread::currentThread();
        QThread::msleep(100);
        m_result = value * 2;
    }

    int getResult() const {
        return m_result;
    }

private:
    int m_result = 0;
};

// 【invokeMethod vs 信号槽】
// | 特性 | invokeMethod | 信号槽 |
// |------|-------------|--------|
// | 代码量 | 少 | 需要声明信号/槽 |
// | 类型安全 | 否（字符串） | 是（编译时） |
// | 灵活性 | 高 | 中 |
// | 推荐场景 | 动态调用 | 一般情况 |
```

```cpp
        QThread* thread = QThread::create([this]() {
            // 【危险】在工作线程更新 UI
            label_->setText("Updated from worker thread!");
            // 可能导致：
            // 1. 崩溃
            // 2. 界面不刷新
            // 3. Qt 警告："QWidget: Cannot create a widget without a parent"

    }

};

class GoodExample : public QWidget {
    Q_OBJECT
public:

        label_ = new QLabel("Hello", this);

        Worker* worker = new Worker;
        QThread* thread = new QThread;
        worker->moveToThread(thread);

        // 【正确】通过信号槽更新 UI
        connect(worker, &Worker::updateRequest, this, &GoodExample::updateLabel);

        connect(thread, &QThread::started, worker, &Worker::doWork);
        connect(thread, &QThread::finished, worker, &QObject::deleteLater);
        connect(thread, &QThread::finished, thread, &QObject::deleteLater);

        thread->start();
    }

    void updateLabel(const QString& text) {
        // 【重点】这个槽在主线程执行

    }

private:
    QLabel* label_;
};
```

```cpp
class FileProcessingApp : public QWidget {
    Q_OBJECT
public:
    FileProcessingApp(QWidget* parent = nullptr) : QWidget(parent) {
        setupUI();

    void setupUI() {

        QPushButton* browseButton = new QPushButton("Browse", this);
        connect(browseButton, &QPushButton::clicked, this, &FileProcessingApp::browseFile);
        fileLayout->addWidget(fileEdit_);
        fileLayout->addWidget(browseButton);
        layout->addLayout(fileLayout);

        // 进度条
        progressBar_ = new QProgressBar(this);
        progressBar_->setRange(0, 100);
        layout->addWidget(progressBar_);

        // 日志
        logEdit_ = new QTextEdit(this);
        logEdit_->setReadOnly(true);
        layout->addWidget(logEdit_);

        // 按钮
        QHBoxLayout* buttonLayout = new QHBoxLayout;
        startButton_ = new QPushButton("Start", this);
        cancelButton_ = new QPushButton("Cancel", this);
        cancelButton_->setEnabled(false);
        connect(startButton_, &QPushButton::clicked, this, &FileProcessingApp::startProcessing);
        connect(cancelButton_, &QPushButton::clicked, this, &FileProcessingApp::cancelProcessing);
        buttonLayout->addWidget(startButton_);
        buttonLayout->addWidget(cancelButton_);
        layout->addLayout(buttonLayout);
    }

    void browseFile() {
        QString filename = QFileDialog::getOpenFileName(this, "Select File");
        if (!filename.isEmpty()) {
            fileEdit_->setText(filename);
        }
    }

    void startProcessing() {
        QString filename = fileEdit_->text();
        if (filename.isEmpty()) {
            QMessageBox::warning(this, "Warning", "Please select a file first.");
            return;
        }

        // 创建工作线程
        workerThread_ = new QThread(this);
        fileProcessor_ = new FileProcessor(filename);
        fileProcessor_->moveToThread(workerThread_);

        // 连接信号
        connect(workerThread_, &QThread::started, fileProcessor_, &FileProcessor::process);
        connect(fileProcessor_, &FileProcessor::progressChanged, this, &FileProcessingApp::updateProgress);
        connect(fileProcessor_, &FileProcessor::logMessage, this, &FileProcessingApp::appendLog);
        connect(fileProcessor_, &FileProcessor::finished, this, &FileProcessingApp::processingFinished);
        connect(fileProcessor_, &FileProcessor::finished, workerThread_, &QThread::quit);
        connect(workerThread_, &QThread::finished, fileProcessor_, &QObject::deleteLater);
        connect(workerThread_, &QThread::finished, this, [this]() {
            workerThread_ = nullptr;
            fileProcessor_ = nullptr;
        });

        // 更新 UI 状态
        startButton_->setEnabled(false);
        cancelButton_->setEnabled(true);
        progressBar_->setValue(0);
        logEdit_->clear();

        workerThread_->start();
    }

    void cancelProcessing() {
        if (fileProcessor_) {
            fileProcessor_->cancel();
        }
    }

public slots:
    void updateProgress(int value) {
        progressBar_->setValue(value);
    }

    void appendLog(const QString& message) {
        logEdit_->append(message);
    }

    void processingFinished() {
        startButton_->setEnabled(true);
        cancelButton_->setEnabled(false);
        QMessageBox::information(this, "Complete", "Processing finished!");
    }

private:
    QLineEdit* fileEdit_ = nullptr;
    QProgressBar* progressBar_ = nullptr;
    QTextEdit* logEdit_ = nullptr;
    QPushButton* startButton_ = nullptr;
    QPushButton* cancelButton_ = nullptr;

    QThread* workerThread_ = nullptr;
    FileProcessor* fileProcessor_ = nullptr;
};

class FileProcessor : public QObject {
    Q_OBJECT
public:
    FileProcessor(const QString& filename, QObject* parent = nullptr)
        : QObject(parent), filename_(filename) {}

    void process() {
        emit logMessage(QString("Processing: %1").arg(filename_));

        // 模拟文件处理
        QFile file(filename_);
        if (file.open(QIODevice::ReadOnly)) {
            qint64 totalSize = file.size();
            qint64 processed = 0;

            while (!file.atEnd() && !cancelled_) {
                QByteArray chunk = file.read(1024 * 1024);  // 1MB chunks
                processed += chunk.size();

                // 处理数据...
                QThread::msleep(50);

                // 发射进度
                int percent = static_cast<int>((processed * 100) / totalSize);
                emit progressChanged(percent);
            }

            file.close();
        }

        emit logMessage(cancelled_ ? "Processing cancelled!" : "Processing complete!");
        emit finished();
    }

    void cancel() {
        cancelled_ = true;
    }

signals:
    void progressChanged(int percent);
    void logMessage(const QString& message);
    void finished();

private:
    QString filename_;
    std::atomic<bool> cancelled_{false};
};
```

## 6.8 多线程面试重点总结

### 常见面试问题

**1. Qt 线程安全保证是什么？**
```
- GUI 操作必须在主线程执行
- 信号槽跨线程时自动使用 QueuedConnection
- 参数通过事件队列传递，自动复制
- 但共享数据仍需手动同步（使用 QMutex）
```

**2. 信号槽跨线程的实现原理？**
```
1. 检测发送者和接收者是否在同一线程
2. 如果跨线程，使用 QueuedConnection
3. 参数被复制（通过 QMetaType）
4. 创建 QMetaCallEvent
5. 投递到接收者线程的事件队列
6. 接收者线程处理事件时调用槽函数
```

**3. 如何避免死锁？**
```
1. 始终以相同的顺序获取多个锁
2. 使用 QMutexLocker（RAII）
3. 避免嵌套锁
4. 使用 QTryMutexLocker 尝试获取锁
5. 不要在持有锁时等待其他线程
```

**4. QThread::run() 和 moveToThread 的区别？**
```
| 特性 | 重写 run() | moveToThread() |
|------|-----------|---------------|
| 对象所在线程 | 主线程 | 工作线程 |
| 代码位置 | run() 方法 | 任意槽函数 |
| 灵活性 | 低（只有一个run）| 高（多个槽）|
| 推荐度 | 不推荐 | 推荐 |
```

**5. QtConcurrent 的使用场景？**
```cpp
// 【面试重点】QtConcurrent 适用于简单任务
#include <QtConcurrent>

void runConcurrent() {
    // 在线程池中运行函数
    QFuture<int> future = QtConcurrent::run([]() {
        QThread::sleep(1);
        return 42;
    });

    // 阻塞等待结果
    int result = future.result();
    qDebug() << "Result:" << result;

    // 使用 QFutureWatcher 非阻塞等待
    QFutureWatcher<int>* watcher = new QFutureWatcher<int>;
    connect(watcher, &QFutureWatcher<int>::finished, [&]() {
        qDebug() << "Async result:" << watcher->result();
        watcher->deleteLater();
    });
    watcher->setFuture(future);
}

// 【适用场景】
// - 简单的并行计算
// - 不需要复杂的线程管理
// - 不需要频繁通信

// 【不适用场景】
// - 长时间运行的任务
// - 需要频繁通信的任务
// - 需要精细控制的场景
```

**6. 子线程更新 UI 的后果？**
```
可能的后果：
1. 崩溃（最常见）
2. 界面不刷新
3. 绘制错误
4. Qt 警告消息

正确做法：
- 使用信号槽
- Qt 自动将槽调用转发到主线程
- 或使用 QMetaObject::invokeMethod + Qt::QueuedConnection
```
